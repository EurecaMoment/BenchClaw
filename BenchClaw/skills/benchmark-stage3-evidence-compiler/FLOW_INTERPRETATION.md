# FLOW_INTERPRETATION

我对手绘 Stage3 图的解析如下：

1. 左侧 `15/16/17` 是 Stage2 的三个终端产物：真实图、已有 benchmark、仿真器多模态 + GT。
2. 图中间的 `统一格式 -> data-juicer -> 半监督标注/GT` 是共同处理逻辑，但不应实现为单一串行瓶颈。为了让 Opencode 更稳定执行，本包把它展开为三条分支上的同构操作：
   - 真实图分支：15 -> 21 -> 24 -> 18
   - 旧 benchmark 分支：16 -> 22 -> 25 -> 19
   - 仿真器分支：17 -> 23 -> 26 -> 20
3. 右侧/下方的 YOLOE、SAM3、Depth Anything 3、小模型路由等视为半监督标注工具注册与执行契约，对应节点 `27`。
4. `18/19/20` 是 Stage3 的三个最终输出：
   - `18`: 真实图 + 半监督 GT；
   - `19`: benchmark 图 + 官方标签/QA + 半监督 GT；
   - `20`: 仿真器 + privileged GT。
5. `20` 不依赖 `27`，因为仿真器 GT 的可信度优先级高于半监督工具输出。

关键原则：**三类数据源要并行处理，不能写成串行链。**
