
# 统一模板系统设计逻辑

## 1. 为什么要统一

Stage1 和 Stage4 本质上不是两个孤立阶段。Stage1 决定“评测什么”和“依据什么数据能测”，Stage4 决定“如何把这个评测设计落成可评分题目”。如果两者使用不同的能力名称、题型定义或指标口径，后续评测集会出现三类问题：

1. Stage1 写了能力，但 Stage4 找不到可实例化模板。
2. Stage4 生成了题目，但 Stage1 没有说明它服务哪个能力维度。
3. 指标看似完整，但缺少 GT 字段、证据引用或过滤规则。

因此本包采用一条贯通链：

```text
能力维度 C → 作答形式 F → 模板 T → GT 字段 G → eval item E → 指标 M → 质量门 Q
```

## 2. 来自用户资料的基础

本包大量吸收了用户提供资料中的三类内容：

- 12 个 benchmark 模板卡片：用于定义任务范式、能力来源、指标口径和常见 GT 依赖。
- 100 条具身空间图文评测模板：作为统一模板库的主体。
- qa1 episode 的问题、tracker 和图像：作为 evidence-grounded 实例化样例。

整合时没有把原资料作为 `source_preserved` 备份目录塞进包里，而是将有用信息转成正式的参考卡片、模板库、schema、示例、指标和质量门。

## 3. 本包的核心判断

具身空间图文评测最容易失败的地方不是“模板数量不够”，而是模板与 GT、证据、指标之间断裂。因此本包把每条模板都看成一个可验证函数：

```text
instantiate(template, sample_gt, evidence) -> eval_item(question, options, answer, metric, evidence_ref)
```

只有当这个函数可执行、可复现、可评分时，模板才允许进入 Stage4。

## 4. Stage1 和 Stage4 的分工

| 环节 | Stage1 使用方式 | Stage4 使用方式 |
|---|---|---|
| Benchmark 参考 | 选择类似任务范式，说明设计依据 | 参考题型、指标、GT 字段，不照搬任务 |
| 能力维度 | 确定评测范围与覆盖矩阵 | 给每道题绑定 capability_id |
| 题型 | 判断哪些作答形式适合自动评分 | 固定 answer_format 与解析方式 |
| 模板 | 形成可实例化需求 | 绑定样本、候选项、答案和证据 |
| 指标 | 形成指标草案 | 实现评分代码与聚合报告 |
| 质量门 | 说明准入条件 | 执行过滤、去歧义和 schema check |

## 5. 设计取舍

- 优先保留能自动评分、GT 依赖清楚、证据链完整的题型。
- 对开放问答、动作序列、导航执行类题目保留参考价值，但在静态图文 QA 场景中降级为辅助或扩展项。
- 对 2D/3D/时序/容器/可见性等题目保留模板主体，因为它们与用户给出的 100 条模板和 qa1 样例最匹配。
- 对每条旧模板中的“人工检查要点”不删除，而是转为质量门与过滤规则。
