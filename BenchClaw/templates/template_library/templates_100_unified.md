# 统一题目模板库（100 条）

本文件是统一模板包的核心模板库。其主体大量来自用户提供的《具身空间智能图文评测模板库（100条）》，并在开头增加统一使用约束，使其同时服务 Stage1 的设计论证和 Stage4 的题目实例化。

## 统一字段解释

| 原字段 | 统一解释 | Stage1 用途 | Stage4 用途 |
|---|---|---|---|
| ID | template_id | 规划模板覆盖 | 绑定每道 eval item |
| 模板问题 | question_template | 判断题面形式 | 实例化 question_text |
| 使用数据类型 | modality/data_type | 判断数据是否支持 | 选择单帧/多帧/GT 输入 |
| 题目作答形式 | answer_format | 规划题型覆盖 | 选择解析器和评分器 |
| 考察能力方向 | capability_id | 能力维度覆盖 | item 归因和分组统计 |
| GT依赖字段 | required_gt_fields | 数据需求清单 | 字段校验与答案生成 |
| 自动评分方式 | scoring_metric | 指标草案 | 实际评分 |
| 人工检查要点 | quality_gate | 风险识别 | 过滤与抽查 |

## 强制约束

1. 模板只有在 GT 字段满足时才能实例化。
2. 模板中的“人工检查要点”必须转成 Stage4 的质量门或过滤规则。
3. 不允许模板答案来自模型常识、语言先验或不可复现推断。
4. 模板实例化后必须写入 `template_id`、`capability_id`、`answer_format`、`scoring.metric`、`evidence_ref`。
5. 对多帧、历史状态、容器关系、可见性题目，必须明确使用的是初始时刻、当前时刻、过去 n 步还是随机历史时刻。

## 附：工程化落地建议

后续如果要将本模板库接入自动合成 pipeline，建议在每个模板条目下进一步补充以下字段：

```yaml
candidate_generation_rule:
answer_generation_rule:
validity_filter:
example_instance:
```

其中 `validity_filter` 建议至少包含以下检查：

1. 候选答案必须唯一；
2. 候选物体必须满足可见性要求；
3. 位置、深度、距离、面积等差异必须超过阈值；
4. 区间类问题的真实值不能落在边界附近；
5. 排序题中相邻项差异不能过小；
6. 题面中的“左/右/前/后”必须明确坐标系；
7. 不可见物体只能用于证据不足判断，不能用于属性识别或视觉描述。
